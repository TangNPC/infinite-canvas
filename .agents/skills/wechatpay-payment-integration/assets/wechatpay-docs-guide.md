## 二、业务目录结构

每个站点下，按业务（产品）划分目录（如 `支付产品/JSAPI支付/`）。每个业务目录下统一组织如下：

| 目录 / 文档 | 内容说明 |
| --- | --- |
| **产品介绍** | 产品功能、适用场景、商户准入、接入前的资质与权限申请等说明 |
| **开发接入准备** | 系统配置、密钥、证书等准备工作 |
| **开发指引** | 接口调用流程、时序、注意事项、最佳实践 |
| **API 列表** | 该产品所有接口的请求/响应参数、字段定义，以及各语言（curl / Java / Go）的请求示例 |
| **常见问题** | 高频问题与排查思路 |
| **错误码** | 接口返回的错误码列表与含义、排查建议 |
| **附录** | 权限申请指引、账号绑定、授权目录配置等补充说明 |

> 不是所有业务都完整包含上述条目，以实际目录为准。

## 三、文档定位流程速查

1. 确认 **角色（普通商户 / 合作伙伴）** 与 **API 版本（V3 / V2）** → 缩小到对应站点目录。
2. 根据用户描述（场景、产品名、接口名等）定位到具体业务目录（如 `支付产品/JSAPI支付/`）；业务模糊时用关键词在站点目录下搜索探路，按命中路径分布判断。
3. 在业务目录下按问题类型选择对应文档（产品介绍 / 开发指引 / 常见问题 / 错误码 / API 列表 等）→ 读取精读。
4. 若需接口请求示例，按 `<接口名>-请求示例-{curl,java,go}-<doc_id>.md` 在该业务的 `API列表/` 下查找。

## 四、单篇文档结构

文件名格式为 `标题-文档ID.md`（如 `产品介绍-4012062524.md`），目录路径为中文（如 `支付产品/JSAPI支付/`）。文件名和路径已包含足够的语义信息，可直接用于判断相关性，无需批量读 `front matter` 做筛选。

每篇 `.md` 文首带有 YAML **front matter**：

```yaml
---
url: "https://pay.weixin.qq.com/doc/v3/merchant/4012062524.md"
title: "产品介绍"
doc_id: 4012062524
---
```

**使用约定**：

- 路径与文件名即语义，无需批量读 `front matter` 来检索
